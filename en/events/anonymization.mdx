A crucial part of a GDPR-compliant setup is **URL Anonymization**. Since we do not collect personally identifiable information (PII), URLs containing personal identifiers (e.g., a User ID, an auth token, or an order number) must be masked or removed.

Our script provides two powerful tools for this: Masking URL paths (`data-anonymize-url`) and stripping query parameters (`data-exclude-params`).

---

## 1. Anonymizing URL Paths

If your website uses URLs like `https://example.com/profile/12345/orders`, the `12345` is a personal User ID. Add the `data-anonymize-url` attribute to your script tag to mask this data locally in the visitor's browser before it is ever sent to us.

### Option A: Auto-Detect Mode (Recommended)

The easiest method is the automatic mode. By setting the value to `true`, the script automatically detects common ID formats and replaces them with `/uuid`.

```html
<script
    defer
    src="https://analytics.bchic.de/script.js"
    data-website-id="YOUR_WEBSITE_ID"
    data-anonymize-url="true"
></script>
```

**What gets detected automatically?**
* **UUIDs:** `/orders/550e8400-e29b...` ➔ `/orders/uuid`
* **Hex hashes (24+ chars):** `/commit/a1b2c3d4e5f6...` ➔ `/commit/uuid`
* **Numeric IDs (18+ digits):** `/invoice/123456789012345678` ➔ `/invoice/uuid`
* **Alphanumeric tokens (20+ chars):** `/session/xK9mP2nQ8rT5...` ➔ `/session/uuid`

### Option B: Custom Templates

If your IDs are shorter (e.g., usernames or short database IDs), you can define custom templates. The script uses a pattern language to mask dynamic parts. You can define multiple rules by separating them with a comma.

```html
<script
    defer
    src="https://analytics.bchic.de/script.js"
    data-website-id="YOUR_WEBSITE_ID"
    data-anonymize-url="/profile/{userId}/*, /order/{orderId}/*!"
></script>
```

#### Template Syntax

| Symbol | Function | Example Output |
| :--- | :--- | :--- |
| `{placeholder}` | Replaces the dynamic part with the placeholder's name. | `/profile/{userId}` |
| *(no wildcard)* | **Exact match:** The path length must match the template exactly. | Input `/a/b/c` is ignored if rule is `/a/b`. |
| `/*` | **Keep sub-paths:** Retains any path segments after the match. | Template `/u/{id}/*` turns `/u/123/settings` ➔ `/u/id/settings` |
| `/*!` | **Drop sub-paths:** Strips everything that comes after the template match. | Template `/u/{id}/*!` turns `/u/123/settings` ➔ `/u/id` |

## 2. Excluding Query Parameters

Personal data often isn't in the path, but hidden after the question mark in the URL (e.g., `?token=abc123` or `?session=xyz`). With the `data-exclude-params` attribute, you can target and remove or empty specific query parameters.

```html
<script
    defer
    src="https://analytics.bchic.de/script.js"
    data-website-id="YOUR_WEBSITE_ID"
    data-exclude-params="/auth/signup:token, /search:q="
></script>
```

### Parameter Rule Syntax

The format is always `[Path-Template]:[Parameter-Name][Action]`. The Path-Template supports the same `{placeholder}` and wildcards (`/*`, `/*!`) described above.

<CardGroup cols={2}>
    <Card title="Delete parameter completely" icon="trash">
        **Format:** `/path:parameter`
        
        The parameter is entirely removed from the URL.
        * **Rule:** `/auth/verify:token`
        * **Original:** `/auth/verify?token=123&lang=en`
        * **Sent as:** `/auth/verify?lang=en`
    </Card>
    <Card title="Empty value (Keep key)" icon="eraser">
        **Format:** `/path:parameter=` (Note the equal sign at the end)
        
        The key is kept, but the value is cleared. Useful if you want to track *that* a search occurred, but not *what* was searched.
        * **Rule:** `/search:q=`
        * **Original:** `/search?q=my+query&sort=asc`
        * **Sent as:** `/search?q=&sort=asc`
    </Card>
</CardGroup>

## Verification & Testing

How do you know if it's working correctly? Check it directly in your browser:

1. Open your browser's **Developer Tools** (F12).
2. Switch to the **Network** tab.
3. Load a page on your website that contains a URL or parameter you want to mask.
4. Look for the tracking request sent to `analytics.bchic.de`.
5. Click on the request and inspect the **Payload**.

<Check>
    The `url` field in the payload should already contain the anonymized form (e.g., `/profile/userId/...` or missing parameters). If this is the case, no personal data ever leaves the user's browser.
</Check>
